## Security Practices

### Smart Contract Development

<CardGroup cols={2}>
  <Card title="OpenZeppelin Base" icon="shield-halved">
    All contracts inherit from OpenZeppelin's audited, battle-tested libraries — AccessControl, Upgradeable, ReentrancyGuard, Pausable.
  </Card>
  <Card title="Foundry Testing" icon="flask-vial">
    Comprehensive test suite using Foundry (forge). Unit tests, fuzz tests, and invariant tests for APMM math and market lifecycle.
  </Card>
  <Card title="Upgrade Safety" icon="lock">
    TransparentUpgradeableProxy with multisig governance. No single key can upgrade contracts. Storage layout compatibility is verified on every upgrade.
  </Card>
  <Card title="Deterministic Deploys" icon="fingerprint">
    Market clones use `Clones.cloneDeterministic()` — the contract address is predictable from the salt, ensuring consistent deployment behavior.
  </Card>
</CardGroup>

### Operational Security

| Practice | Description |
|----------|-------------|
| **Multisig governance** | Admin roles (upgrade, pause) controlled by multisig wallet |
| **Key management** | Private keys stored in hardware security modules (HSMs) |
| **Least privilege** | Each service has only the permissions it needs — the API cannot upgrade contracts, the admin cannot sign user transactions |
| **Monitoring** | Block Listener provides real-time alerting on anomalous on-chain events |
| **Incident playbook** | Documented response procedures for contract pause, fund recovery, and communication |

### Code Quality

## Known Considerations

The following are known design decisions, not vulnerabilities, but important to understand:

<Accordion title="Mock Oracle on testnet">
The staging environment uses `MockOptimisticOracleV3` instead of the real UMA Oracle. This mock oracle auto-resolves assertions without a challenge period. On mainnet, the full UMA Optimistic Oracle with bonding and dispute escalation will be used.
</Accordion>

<Accordion title="Upgradeable proxy pattern">
The PredictionFactory is upgradeable via TransparentUpgradeableProxy. This means the contract logic can be changed by the `UPGRADER_ROLE` holder (multisig). While this enables bug fixes and feature additions, it also means the system is not fully immutable. Existing market clones are not affected by factory upgrades — only new markets use the updated implementation.
</Accordion>

<Accordion title="Paymaster gas sponsorship">
The Paymaster sponsors all user gas, creating a potential grief vector if an attacker submits many transactions to drain the Paymaster's ETH balance. Protections include per-user rate limits, allowlisted operations, and daily sponsorship caps.
</Accordion>

<Accordion title="Admin service direct database access">
The Admin service connects directly to PostgreSQL rather than going through the Market Service gRPC layer. This is a conscious design choice for operational flexibility but means the admin service bypasses gRPC-layer validation. Admin access is protected by RBAC and audit logging.
</Accordion>

## Bug Bounty

A formal bug bounty program will be launched alongside the mainnet deployment. In the meantime, if you discover a security vulnerability:

<Steps>
  <Step title="Do not publicly disclose">
    Please do not open a public issue or discuss the vulnerability in public channels.
  </Step>
  <Step title="Email the security team">
    Send a detailed report to **security@promethex.market** with:
    - Description of the vulnerability
    - Steps to reproduce
    - Potential impact assessment
    - Suggested fix (if any)
  </Step>
  <Step title="Expect a response within 48 hours">
    We will acknowledge receipt and provide an initial assessment within 2 business days.
  </Step>
  <Step title="Coordinated disclosure">
    We will work with you on a responsible disclosure timeline and acknowledge your contribution.
  </Step>
</Steps>
