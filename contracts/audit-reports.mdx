## Security Practices

### Static Analysis

All contracts are continuously analyzed with:

| Tool | Purpose | Integration |
|------|---------|-------------|
| [Slither](https://github.com/crytic/slither) | Vulnerability detection, code quality | CI pipeline |
| [Mythril](https://github.com/Consensys/mythril) | Symbolic execution, bug detection | CI pipeline |
| [Solhint](https://github.com/protofire/solhint) | Linting, style enforcement | Pre-commit hook |

### Testing

| Test Type | Coverage | Framework |
|-----------|----------|-----------|
| Unit tests | All public functions | Foundry (forge test) |
| Fuzz testing | APMM invariant, deposit/withdraw round-trips | Foundry fuzz |
| Integration tests | Full market lifecycle (create → trade → settle → claim) | Foundry |
| Fork tests | UMA oracle integration on Arbitrum fork | Foundry fork mode |

```bash
# Run full test suite
cd prome-contracts/contracts
forge test -vvv

# Run with gas report
forge test --gas-report

# Run fuzz tests (extended)
FOUNDRY_FUZZ_RUNS=10000 forge test --match-test testFuzz
```

### Dependency Management

- All OpenZeppelin contracts pinned to specific versions (`@openzeppelin/contracts-upgradeable@5.x`)
- Foundry dependencies managed via git submodules with pinned commits
- No external dependencies beyond OpenZeppelin and UMA interfaces

## Known Limitations

<AccordionGroup>
  <Accordion title="Oracle liveness window">
    If no one disputes an incorrect assertion within the liveness period, the incorrect outcome becomes final. This is inherent to the optimistic oracle model. Mitigation: run monitoring bots and set appropriate liveness periods.
  </Accordion>
  <Accordion title="Clone upgradeability">
    Existing Prediction clones cannot be individually upgraded. If a critical bug is found in a deployed market, it cannot be patched — only new markets benefit from implementation updates. Mitigation: thorough testing and auditing before mainnet deployment.
  </Accordion>
  <Accordion title="Centralized market creation">
    Market creation is gated by `CREATOR_ROLE`. This is intentional for quality control during the platform's early phase but limits permissionless market creation. Planned: decentralized market creation in a future release.
  </Accordion>
  <Accordion title="Single-owner settlement">
    Only the market owner can call `settle()`. If the owner's key is lost or the owner becomes unresponsive, the market cannot be settled. Planned: emergency settlement mechanisms and multisig ownership.
  </Accordion>
  <Accordion title="Fixed-decimal arithmetic">
    APMM calculations use SD59x18 (signed 59.18-decimal fixed-point). While this provides 18 decimal places of precision, extreme reserve ratios (>1e30:1) may cause rounding errors or overflow. Markets are initialized with balanced reserves to avoid this.
  </Accordion>
</AccordionGroup>

## Responsible Disclosure

If you discover a vulnerability in PrometheX smart contracts:

<Steps>
  <Step title="Do NOT disclose publicly">
    Do not open GitHub issues, tweet, or discuss the vulnerability in public channels.
  </Step>
  <Step title="Contact the team">
    Email **security@promethex.market** with:
    - Description of the vulnerability
    - Steps to reproduce
    - Potential impact assessment
    - Your suggested fix (optional)
  </Step>
  <Step title="Response timeline">
    We aim to acknowledge within **24 hours** and provide a detailed response within **72 hours**. Critical issues will be patched and deployed within **7 days**.
  </Step>
  <Step title="Recognition">
    With your permission, we will credit you in the security advisory and on our Hall of Fame page. Bounty rewards (when the program is active) will be paid in USDC.
  </Step>
</Steps>
